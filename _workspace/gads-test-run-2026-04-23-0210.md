## Confirmed Schema (v23)

### T8 — 캠페인 성과 리포트용

```gaql
SELECT
  campaign.id,
  campaign.name,
  campaign.status,
  campaign.advertising_channel_type,
  segments.date,
  metrics.impressions,
  metrics.clicks,
  metrics.cost_micros,
  metrics.conversions,
  metrics.conversions_value,
  metrics.ctr,
  metrics.average_cpc
FROM campaign
WHERE campaign.advertising_channel_type = 'PERFORMANCE_MAX'
  AND segments.date BETWEEN '2026-04-16' AND '2026-04-23'
ORDER BY segments.date DESC, campaign.id
LIMIT 100
```

**사용처**: 광고주 대시보드 메인 성과 테이블.

```gaql
  segments.product_title,
  segments.product_brand,
```

### T10 — 일별 비용 정산


```gaql
-- 캠페인별 일별 비용 (정산 배치용)
SELECT
  campaign.id,
  campaign.name,
  segments.date,
  metrics.cost_micros,
  metrics.impressions,
  metrics.clicks,
  metrics.conversions_value
FROM campaign
WHERE segments.date BETWEEN '2026-03-24' AND '2026-04-23'
  AND campaign.advertising_channel_type = 'PERFORMANCE_MAX'
ORDER BY segments.date DESC, campaign.id
```

**사용처**: 매일 새벽 배치가 전일 비용을 가져와서 광고주 정산 테이블 적재.

**Timezone 주의사항** (프로덕션 중요):
- `segments.date` 는 **계정 timezone** 기준 날짜로 잘림
- 무신사 서비스는 `Asia/Seoul` — 계정 timezone 도 `Asia/Seoul` 이어야 일자 경계 누락 없음
- 프로덕션 광고주 계정 생성 시 timezone 통일 필수 (체크리스트 항목으로 추가)

## Key Findings

### 1. 모든 metrics/segments 필드 경로가 v23 스키마에 존재

- `segments.date` 일별 분해 OK
- `metrics.cost_micros` 단위 micros (1 KRW = 1,000,000 μ)
- `metrics.ctr`, `metrics.average_cpc` 파생 metric 도 바로 SELECT 가능 (별도 계산 불필요)
- `segments.product_*` 전체 세트 동작 확인

### 2. `customer` 리소스에서 currency_code / time_zone 조회 가능

정산 배치 설계 시 timezone 확인을 **매일 쿼리 내에서** 함께 받아올 수 있음.
외부 설정 파일로 관리할 필요 없고, 광고주 계정별 timezone 이 달라도 쿼리 결과에서 바로 확인.

### 3. custom_attribute0 의 실용 가치

현재 feed 에 값이 없으므로 이 컬럼은 리포트에 표시해도 의미 없음.
**feed 스펙 수정(협력팀 요청) 후에만 "업체별/브랜드별 ROAS" 같은 분해가 가능**.
현아님에게 전달할 피드 수정 요청에 이 점도 함께 포함.
