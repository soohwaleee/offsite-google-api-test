## Key Findings

### 1. 단일 mutate 에 102 ops 여유롭게 가능

Google Ads API 공식 문서상 단일 mutate 의 **최대 ops 는 5,000 개**.
100개 상품 선택 시나리오는 **2% 수준만 사용**하므로 여유롭게 가능.

→ 실무에서 광고주가 **수천 개 상품을 동시에 선택**하는 극단적 케이스도 API 상 무리 없음.
단, 그 이상(예: 5,000+ 개) 에서는 **chunk 분할 필요**.

### 2. Temp resource_name 패턴이 대량 처리에도 안정 동작

```kotlin
val pathBase = "customers/$customerId/assetGroupListingGroupFilters/$assetGroupId"
val rootTemp = "$pathBase~-1"
items.forEachIndexed { idx, itemId ->
    val tempName = "$pathBase~-${idx + 2}"
    // children.parent_listing_group_filter = rootTemp
}
```

- 음수 ID (`~-1`, `~-2`, ...) 를 임시 식별자로 사용
- 같은 mutate 내에서 부모-자식 참조 가능
- Google 이 commit 시점에 실제 ID 로 resolve

### 3. 응답 속도: 100ms 당 ~9 ops 수준

```
1,090 ms / 102 ops = 10.7 ms/op
```

광고주 UI 에서 "100개 상품 선택 → 저장" 버튼 클릭 시 **1~2초 내 응답 가능**.
광고주가 기다림 없이 자연스러운 UX 구현 가능한 수준.
